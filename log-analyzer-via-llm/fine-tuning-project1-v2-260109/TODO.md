#### 代码

1. 我们写的这个叫代码吗？（还可以更好）
2. 代码本身只具备学习价值。对企业，它的主要价值是可再现可运行的系统。
3. 所以代码中应包含自构建方法（所谓安装脚本，至少应支持企业主力 OS。应稍稍探测环境 —— 网络不通不应等发飞书时报错，没有GPU不应等推理时报错）
4. 代码也应是自解释的（依托一些行业惯例）
5. 代码也应是自文档化的（以 [metacpan](https://metacpan.org/pod/AnyEvent::Log) 为例）


#### 文档

1. 如果项目不流转到技术同学之外，则不需要纯文字的文档
2. 所以文档的用户是谁？（是降低非技术同学的沟通成本的工具，因为他们对技术有隐含的恐惧和拒绝，“文档”能卸下他们本能的抵抗）
3. 所以文档应该如何写？此文档非刚刚代码中的“自文档化”（它某种程度上应该是anti-tech的）
4. 理论上，只要我们写的产品有非技术用户（很多时候是市场营销等帮助我们的同学），就都可以写出文档（想想 Linux ）
